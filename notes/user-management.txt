Context
The qabot UI currently operates as a single-user tool with zero authentication — all API routes are open, WebSocket broadcasts to all clients, and data stores have no user context. This feature adds local multi-user support with user accounts, role-based access control (admin/tester/viewer), and a comprehensive audit log — all without any cloud dependency.
Why this matters: Teams sharing a qabot instance (e.g., on a shared dev server or via VPN) currently have no way to track who ran what, no way to restrict destructive operations to senior team members, and no accountability trail. This feature transforms qabot from a personal tool into a team-ready platform.

Feature Scope & What's Included
Core Features

User Accounts — Local accounts with username/password, display name, active/disabled status
Role-Based Access Control (RBAC) — Admin, Tester, Viewer roles with hierarchical permissions
Audit Log — Every significant action tracked with user, timestamp, action type, resource, and details
First-Time Setup Wizard — Guided creation of initial admin account on fresh installations
Session Management — JWT-based token auth for both REST API and WebSocket connections
User Management UI — Admin page to create/edit/disable users, reset passwords, manage sessions
Audit Log UI — Filterable, paginated audit log viewer with export capability

Additional Enhancements

User Avatar in Header — Logged-in user with initial letter avatar + dropdown menu (profile, change password, logout)
Role-Aware UI — Buttons/actions dynamically show/hide based on user role (viewers see read-only mode)
Activity Attribution — Dashboard activity feed shows WHO performed each action
RoleVisible Component — Reusable wrapper to conditionally render UI elements by role


Architecture Decisions
Zero New Dependencies
All functionality built with Node.js built-ins + existing packages:

Password hashing: node:crypto scryptSync (built-in, no bcrypt needed)
JWT tokens: node:crypto HMAC-SHA256 (built-in, no jsonwebtoken needed)
SQLite storage: better-sqlite3 (already a dependency)
ID generation: generateId() (already exists in src/utils/)

Storage Strategy

Users + Sessions → SQLite (~/.qabot/users.db) — needs indexed lookups, atomic operations, concurrent access
Audit Log → SQLite (~/.qabot/audit.db) — needs efficient querying, filtering, pagination, rotation
JWT Secret → File (~/.qabot/auth-secret.key) — 64-byte random hex, generated once

Existing Data Remains Shared
Test results, cloud config, chat history, API collections — all remain global/shared (not user-scoped). Teams share one project workspace; the audit log tracks WHO did what.

Implementation Plan
Phase 1: Backend Auth Infrastructure (4 new files)
File: src/ui/auth/token-manager.ts

JWT signing/verification using node:crypto HMAC-SHA256
Generates + persists signing secret at ~/.qabot/auth-secret.key
Token payload: { sub, jti, role, username, iat, exp } (24h expiry default)
sign(payload) and verify(token) methods

File: src/ui/store/user-store.ts

SQLite database at ~/.qabot/users.db
Tables: users (id, username UNIQUE COLLATE NOCASE, display_name, password_hash, role, is_active, timestamps) + sessions (id/jti, user_id, expires_at, revoked)
Password hashing: scryptSync with random 16-byte salt, format salt:hash
Methods: CRUD users, create/revoke sessions, hasAnyUsers(), verifyPassword(), getAdminCount()
Following existing GitCacheStore / HealingStore SQLite patterns

File: src/ui/auth/permissions.ts

Role hierarchy: admin(3) > tester(2) > viewer(1)
hasPermission(userRole, requiredRole) helper
Route permission map: maps HTTP method + path pattern to minimum role
Declarative config approach — not scattered across route files

File: src/ui/auth/auth-middleware.ts

AuthenticatedRequest interface extending Express Request with user property
Core middleware: extracts Bearer token, verifies JWT, checks session not revoked, attaches req.user
Public paths whitelist: /api/auth/login, /api/auth/setup, /api/auth/status
Non-API paths (static files) pass through — SPA handles auth client-side
Setup mode: when hasAnyUsers() === false, API routes return { setupRequired: true }
Role enforcement using the permission map from permissions.ts

Phase 2: Auth API Routes (1 new file)
File: src/ui/routes/api-auth.ts
Public endpoints (no auth):

GET /api/auth/status — { setupRequired, authenticated, user? }
POST /api/auth/setup — Create first admin (only when zero users)
POST /api/auth/login — Authenticate, return JWT token + user info

Authenticated endpoints:

POST /api/auth/logout — Revoke current session
POST /api/auth/change-password — Change own password
GET /api/auth/me — Get current user profile

Admin-only endpoints:

GET /api/auth/users — List all users
POST /api/auth/users — Create new user
PUT /api/auth/users/:id — Update user (displayName, role, isActive)
DELETE /api/auth/users/:id — Deactivate user (soft delete)
POST /api/auth/users/:id/reset-password — Admin reset password
POST /api/auth/users/:id/revoke-sessions — Force logout a user

Edge cases: last-admin protection (cannot delete/demote the only active admin), password min 6 chars.
Phase 3: Audit Log Backend (2 new files)
File: src/ui/store/audit-log-store.ts

SQLite at ~/.qabot/audit.db, indexed on timestamp, user_id, action, resource
Action categories: auth.*, user.*, run.*, file.*, settings.*, ai.*, git.*
log(entry) — synchronous write, non-blocking, silent on failure
query(filter) — paginated with filters (userId, action prefix, resourceType, date range)
prune(olderThanDays=90) — runs on startup + every 24h interval

File: src/ui/routes/api-audit.ts

GET /api/audit — Query with filters (admin only)
GET /api/audit/export — Download full log as JSON
GET /api/audit/stats — Summary (actions per user, login counts)

Edge Cases Handled

Last admin protection — Cannot delete/demote the only active admin (checked server-side)
Session expiry — 401 response auto-clears token and redirects to login
Concurrent sessions — Allowed; admin can force-revoke all sessions for a user
First-run migration — Existing installations see setup wizard; all existing data preserved
WebSocket auth — Token passed as query param; invalid token gets 4001 close code
Password reset — Admin-only; no email (local tool)
Upload auth — uploadFiles() uses raw fetch, needs Authorization header added separately


Verification Plan

Fresh install: No users.db → setup wizard → create admin → auto-login → dashboard
Login/logout: Logout → login page → wrong password → error → correct → dashboard
Tester role: Login as tester → can run tests, edit files → cannot access Settings modifications, Users, Audit
Viewer role: Login as viewer → can see dashboards/results → cannot start runs, edit files, commit
Last admin: Try to delete/demote only admin → rejected with error
Session expiry: Expire session in DB → next API call → 401 → redirect to login
WebSocket: No token → rejected (4001) → with token → receives broadcasts
Audit log: Login, run tests, edit file, git commit → audit page shows all with correct user
Concurrent sessions: Two browsers → both work → admin force-revokes → both redirect to login
Build: pnpm build && pnpm run build:ui — both pass
