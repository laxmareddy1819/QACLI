Context
The user's question: "How will aiHealing and visionHealing work — we're not capturing test name and step name — what context are we sending to resolve?"
Current state: AI and Vision healing work, but with minimal context to the LLM. The LLM receives a broken selector + stored fingerprint attributes + optional screenshot, but has no idea:

What test is running
What step/action failed (click? fill? hover?)
What scenario/describe block this is in
What the test file path is
What the current DOM looks like

This significantly reduces the LLM's ability to make accurate suggestions.

Gap Analysis — What's Sent vs What's Missing
Phase 2: AI Text-Based Healing (resolveAI → AIHealer.heal())
Currently sent to LLM:

✅ Broken selector string
✅ Stored fingerprint (~13 attributes: tagName, id, testId, className, ariaLabel, etc.)
✅ Optional DOM context (via domSnapshot param — but no adapter actually sends this)
❌ Test name (e.g., "Login with valid credentials")
❌ Step name (e.g., "When I click the login button")
❌ Scenario/describe name (e.g., "Login functionality")
❌ Action type (click, fill, hover, waitFor, etc.)
❌ Test file path (e.g., src/steps/login.steps.ts)
❌ Error message (available in request but NOT passed to AIHealer)

Phase 3: Vision-Based Healing (resolveVision → AIHealer.healWithVision())
Currently sent to LLM:

✅ Screenshot (full page PNG)
✅ Page URL
✅ Broken selector
⚠️ Element description (only 6 of 15 fingerprint fields: tag, text, aria-label, type, placeholder, class)
❌ Stored fingerprint from server — vision endpoint uses fingerprint || { tagName: 'unknown' } from request, NOT the rich stored fingerprint from the DB
❌ Test/step/scenario context (same as AI)
❌ Action type (same as AI)
❌ Error message (not even accepted by /api/heal/vision)
