Context
qabot is a local/on-premise AI-powered test automation CLI with a React dashboard. The key value proposition vs cloud competitors (Testim, Mabl, Applitools, Katalon): runs locally, works with any framework, AI-powered, no vendor lock-in, full data privacy.
Current state: 36 tools, 10 modules, 8 dashboard pages, multi-LLM support, self-healing, cloud grid integration, CI/CD generation, scheduling, failure analysis. A comprehensive foundation — now we add the features that make it a commercial-grade platform.

TIER 1: QUICK WINS (1-3 days each, high value)
1.1 Saved Run Configurations (1 day)
Save frequently-used run commands as named presets (command, env vars, cloud provider, framework). One-click re-run.

Why: Teams run the same suites daily. Removes friction.
How: New ~/.qabot/run-presets.json store (follows CloudConfigStore pattern), CRUD API, preset dropdown on Runner page.

1.2 Environment Manager (2 days)
Define named environments (Dev, Staging, QA, Prod) with per-env base URLs, credentials, env vars, feature flags. Select environment when running tests.

Why: Every project needs multi-environment support. Currently manual via .env files.
How: New ~/.qabot/environments.json store, CRUD API, env selector on Runner page. Runner injects selected env vars at runtime.

1.3 Run Comparison / Side-by-Side Diff (2 days)
Compare two test runs: which tests changed status (pass→fail, fail→pass), duration regressions, new/removed tests.

Why: #1 question after a failure: "What changed since last run?"
How: New GET /api/results/compare?run1=X&run2=Y, match tests by name across StoredRun objects, compute status transitions. New RunCompare.tsx component with color-coded diff.

1.4 Test Duration Heatmap (1-2 days)
Visual heatmap of test execution times across runs. Color intensity = duration. Spot tests getting slower.

Why: Slow tests are hidden CI cost. Heatmaps make trends instantly visible.
How: New GET /api/results/duration-matrix, CSS grid or Recharts heatmap component. Data already exists in testHistory.

=======
1.5 Exportable PDF/HTML Reports (2-3 days)  === Done
Self-contained HTML test report with summary stats, charts, failure details, screenshots. One-click download.

Why: QA managers share results with stakeholders who don't use the dashboard. Top-3 requested feature in test management surveys.
How: New report-generator.ts service producing inline-CSS HTML. GET /api/results/runs/:id/report?format=html. "Export" button on Results page.
============

1.6 Notifications — Slack/Teams/Webhooks (2-3 days)
Send alerts when tests fail, scheduled runs complete, or flaky rate exceeds threshold. Slack webhooks, Teams webhooks, generic HTTP webhooks.

Why: Table-stakes for CI-adjacent tools. Critical for scheduled run workflows.
How: New notification-store.ts + notification-service.ts. Hook into runner completion (api-runner.ts) and scheduler completion. Settings page for webhook config.


TIER 2: CORE DIFFERENTIATORS (1-2 weeks each)
2.1 Visual Regression Testing (1.5-2 weeks)
Capture baseline screenshots, compare future runs with pixel diffing + structural comparison. Highlight diffs, approve/reject changes. All processing local — no data leaves the machine.

Why: Applitools charges $10K+/year. No tool offers visual regression on-premise. This alone justifies qabot for privacy-conscious enterprises.
How: New src/visual/ module. pixelmatch library for pixel diff (pure JS, no native deps). BaselineStore at ~/.qabot/visual-baselines/. New visual_compare tool for LLM usage. New /visual dashboard page with baseline gallery, side-by-side slider, approve/reject workflow.

2.2 Accessibility Testing — WCAG 2.1/2.2 (1-1.5 weeks)
Scan pages for WCAG compliance. Report violations with severity, affected elements, WCAG criterion, and AI-powered fix suggestions.

Why: WCAG compliance is an increasing legal requirement (ADA, EU EAA). No local AI tool integrates accessibility testing into an automation workflow.
How: Inject axe-core (MIT, ~250KB) via page.evaluate(). New accessibility_scan tool. AI enrichment for fix suggestions via LLM. New /accessibility dashboard page with violation list, severity charts, element highlight.

===============
2.3 API Testing Module (2 weeks) === Done
REST/GraphQL request builder, response validation, environment variable substitution, request chaining, contract testing (OpenAPI schema validation).

Why: API testing is the most common automation type. Currently qabot only supports browser testing. Adding API makes it a complete platform.
How: New src/api-testing/ module. New tools: api_request, api_validate_schema. New ~/.qabot/api-collections.json store (Postman-like collections). New /api-testing dashboard page with request builder, response viewer, collection manager.
=============

2.4 Test Impact Analysis (1.5-2 weeks)
Analyze git diff → determine which tests to run. Three strategies: import graph tracing, naming convention matching, historical failure correlation.

Why: Reduces CI time by 60-80%. Killer enterprise feature. Running locally = no code leaves the machine.
How: New impact-analyzer.ts service. Parse imports from test files. Match by convention (src/auth/login.ts → tests/auth/login.spec.ts). Correlate with testHistory. New POST /api/impact/analyze, impact panel on Runner page with "Run only affected" option.

2.5 Autonomous Exploration Agent (2+ weeks)
AI agent that autonomously explores a web app, discovers pages, fills forms, clicks buttons, and documents bugs. No scripts required.

Why: The "holy grail" of AI testing. qabot already has the full browser stack + LLM integration — needs exploration strategy layer.
How: New exploration-agent.ts service. Specialized system prompt for exploration mode. Bug detection: console errors, HTTP 4xx/5xx, broken links, form validation issues. Real-time WS streaming of exploration progress. New /explore page with coverage map, discovered pages, and bugs found.


TIER 3: ENTERPRISE FEATURES (2-4 weeks each)
3.1 Quality Gate / Release Readiness Score (1-1.5 weeks)
AI-computed quality score (0-100): pass rate, trend direction, flaky density, failure severity, regression count. Display as "Release Ready" / "Needs Attention" / "Do Not Release".

Why: Leaders need ONE answer: "Is this build safe to release?" No tool does this holistically.
How: New quality-gate.ts service. Configurable weights per component. Badge on Dashboard + Results. CLI command qabot quality-gate for CI integration (exit code based on score).

3.2 Intelligent Test Prioritization (1 week, builds on 2.4)
Rank tests by risk per commit: historical failure rate, change proximity, test age, flakiness, complexity. Run highest-risk tests first.

Why: Find bugs faster when CI time is limited. Combined with 2.4, a powerful CI optimization engine.
How: New test-prioritizer.ts. Risk scoring model on existing data. "Smart Order" toggle on Runner page.

3.3 Plugin System (2-3 weeks)
Extensible architecture: custom tools, parsers, reporters, dashboard widgets via plugins.

Why: Enterprise teams have unique toolchains. Plugins make qabot adaptable without forking.
How: Plugin discovery in ~/.qabot/plugins/. Manifest: qabot-plugin.json. Lifecycle: activate(context)/deactivate(). Context provides registerTool(), registerParser(), registerRoute().

3.4 Multi-Project Support (2 weeks)
Manage multiple test projects from one dashboard. Cross-project analytics, switch between projects.

Why: Enterprise teams have 5-20 test projects. Single-project is unsustainable.
How: Project registry store. Namespace all stores by project ID: ~/.qabot/projects/{id}/. Project selector in header.

3.5 Full Privacy Mode (1-1.5 weeks)
Verified mode where ALL AI runs through local LLMs (Ollama/LM Studio). Visual indicator: "Full Privacy Mode — No data leaves this machine." Block cloud LLM providers.

Why: THE key differentiator for regulated industries (healthcare, finance, defense, government).
How: Config flag privacyMode: 'full'|'standard'. Block cloud LLMs, disable external network calls. Privacy badge in header. Network audit logging.

KEY ARCHITECTURAL NOTES

Storage Evolution: Current JSON stores work for single-project. For Tier 3+ (multi-project, multi-user), migrate to SQLite as primary store (already a dependency).
Plugin-First Design: Build 3.3 Plugin System before Tier 3 features so visual regression, accessibility, and API testing can be first-party plugins — proves the architecture.
Sidebar Growth: Adding 4-5 new routes will push sidebar to 13+ items. Add collapsible sections before Tier 2.
LLM Cost Tracking: Features like autonomous exploration make heavy LLM calls. Track token usage with cost estimates. Privacy mode with local LLMs = zero incremental cost (selling point).
New Routes: /visual, /accessibility, /api-testing, /explore, /quality — each follows existing pattern in src/ui/routes/ + ui-app/src/components/.
