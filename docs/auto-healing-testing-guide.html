Prerequisites & Setup

# 1. Build everything
pnpm run build:ui && pnpm build

# 2. Start the qabot UI server (port 3700)
node bin/qabot.js /buildUI

# 3. Open browser
# Navigate to http://localhost:3700
Note: The healing SQLite database is stored at ~/.qabot/healing.db. If you want a clean test, delete this file before starting. On Windows the path is C:\Users\{username}\.qabot\healing.db.

Part 1: CLI Testing (REPL Commands)

All tests below run inside the qabot REPL (interactive mode).

Test 1.1 — /heal (No Subcommand)

/heal
Expected: Shows healing status summary — fingerprint count, event stats, success rate. If no data exists yet, shows zeros.

Test 1.2 — /heal status

/heal status
Expected: Formatted table showing: • Fingerprints stored count • Healing events (total / healed / failed) • Success rate % • Avg confidence, avg duration • Strategy breakdown (if events exist) • Framework breakdown (if events exist) • List of injected projects (if any)

Test 1.3 — /heal adapters

/heal adapters
Expected: Lists all 6 supported adapter combinations: 1. Playwright — TypeScript/JavaScript 2. Selenium — Java 3. Selenium — Python 4. Cypress — TypeScript 5. WebdriverIO — TypeScript 6. Selenium — C#

Test 1.4 — /heal inject (Auto-detect Framework)

# First, create a dummy Playwright project:
mkdir /tmp/test-heal-project
cd /tmp/test-heal-project
npm init -y
npm install @playwright/test
# In qabot REPL:
/heal inject /tmp/test-heal-project
Expected: • Auto-detects Playwright + TypeScript • Creates 2 files: tests/helpers/healing-client.ts and tests/fixtures/healing.ts • Shows integration instructions (import the fixture, use test() instead of default) • Shows configuration (server URL, confidence threshold, AI enabled)

Test 1.5 — Verify Injection was Recorded

/heal status
Verify: Shows the injected project in the list with path, framework: playwright, language: typescript, status: active.

Test 1.6 — /heal report

/heal report 7
/heal report 30
/heal report 90
Expected: Each shows analytics for the specified time window (likely empty/zero if fresh database).

Test 1.7 — /heal remove

/heal remove /tmp/test-heal-project
Expected: Marks the injection as removed. /heal status should reflect the change.

Part 2: API Testing (HTTP Bridge)

These test the healing API endpoints that external test frameworks call. Use curl from a terminal while qabot UI server is running on port 3700.

Test 2.1 — Store a Fingerprint (Baseline)

curl -s -X POST http://localhost:3700/api/heal/fingerprint \
  -H "Content-Type: application/json" \
  -d '{
    "selectorKey": "css:#login-button",
    "url": "https://example.com/login",
    "fingerprint": {
      "tagName": "button",
      "id": "login-button",
      "testId": "login-btn",
      "className": "btn btn-primary",
      "ariaLabel": "Sign In",
      "textContent": "Sign In",
      "type": "submit"
    },
    "framework": "playwright"
  }' | jq .
Expected: { "id": "...", "stored": true }

Test 2.3 — Resolve a Broken Selector (Main Healing Endpoint)

curl -s -X POST http://localhost:3700/api/heal/resolve \
  -H "Content-Type: application/json" \
  -d '{
    "selector": "#login-btn-old",
    "selectorStrategy": "css",
    "fingerprint": {
      "tagName": "button",
      "id": "login-btn-old",
      "testId": "login-btn",
      "className": "btn btn-primary updated",
      "ariaLabel": "Sign In",
      "textContent": "Sign In",
      "type": "submit"
    },
    "pageUrl": "https://example.com/login",
    "framework": "playwright",
    "language": "typescript"
  }' | jq .
Expected: Returns healed selector: 
{
  "healed": true,
  "selector": "[data-testid=\"login-btn\"]",
  "selectorStrategy": "css",
  "confidence": 0.90,
  "strategy": "fingerprint",
  "durationMs": ...,
  "aiUsed": false,
  "candidates": [...]
}

Test 2.4 — Resolve Without Stored Fingerprint (Should Fail)

curl -s -X POST http://localhost:3700/api/heal/resolve \
  -H "Content-Type: application/json" \
  -d '{

    "pageUrl": "https://example.com/unknown",
    "framework": "cypress"
  }' | jq .
Expected: { "healed": false, ... } — no stored fingerprint to match against.

Part 3: UI Testing (Browser)

Open http://localhost:3700 in your browser. If auth is configured, log in first.

Test 3.1 — Main Dashboard → HealingCard

  1. Navigate to / (Dashboard)
  2. Verify: A "Self-Healing" card appears in the bottom section
  3. Verify: Shows "Elements tracked" count and "Success rate" percentage
  4. Verify: Card is clickable — clicking navigates to /healing
Note: If no healing data exists, the card may be hidden (returns null when available: false).

Test 3.2 — Healing Dashboard → Overview Tab

  1. Navigate to /healing (click "Healing" in the sidebar or the dashboard card)
  2. Verify: Page title "Self-Healing Dashboard" with Heart icon
  3. Verify: Day filter dropdown (7 / 30 / 90 days) in top-right corner
  4. Verify: 6 overview stat cards at the top:
    • Total Events (blue Activity icon)
    • Success Rate (color-coded: green ≥80%, amber ≥50%, red <50%)
    • Avg Confidence (brand color)
    • Avg Duration (cyan Clock icon)
    • AI Usage % (violet Sparkles icon)
    • Active Projects (pink FolderOpen icon)
  5. Verify: 5 tabs visible: Overview | Events | Fingerprints | Projects | Adapters
  6. Verify: Overview tab shows (when data exists):
    • Healing Trend chart (stacked area: green healed / red failed + purple success % line)
    • Strategy Breakdown bar chart (color-coded per strategy)
    • Framework Breakdown horizontal bar chart
    • Top Failures table (selector, URL, failure count, last seen)
  7. Test day filter: Change from 30 → 7 days → verify data updates
  8. Test empty state: With no data, charts show "No healing data yet" messages

Test 3.3 — Healing Dashboard → Events Tab

  1. Click "Events" tab
  2. Verify: Search bar, Framework dropdown, Result dropdown (All/Healed/Failed), event count
  3. Verify: Table columns: Status | Selector | Healed To | Strategy | Framework | Confidence | Duration | Time
  4. Verify: Status icons: green checkmark for healed, red X for failed
  5. Verify: AI-powered events show purple Sparkles icon next to strategy
  6. Verify: Confidence is color-coded (green ≥0.8, amber ≥0.5, red <0.5)
  7. Test search: Type a selector name → table filters in real-time
  8. Test framework filter: Select "playwright" → only Playwright events shown
  9. Test result filter: Select "Healed" → only successful heals shown
  10. Test pagination: If >25 events, verify prev/next buttons and "Showing X–Y of Z" text

Test 3.4 — Events Tab → Detail Panel

  1. On the Events tab, click any event row
  2. Verify: A 380px side panel opens on the right
  3. Verify panel contents:
    • Header with Healed/Failed badge + AI badge (if applicable) + X close button
    • Selector Change: Original selector (red box) → arrow → Healed selector (green box)
    • Strategy: Name + human-readable description
    • Confidence: Large % number + High/Medium/Low label + progress bar
    • Details grid: URL, Framework, Language, Duration, Timestamp
  4. Verify: Selected row is highlighted in the table (left blue border)
  5. Click another row: Panel updates to show new event details
  6. Click X button: Panel closes, row highlight removed

Test 3.5 — Healing Dashboard → Fingerprints Tab

  1. Click "Fingerprints" tab
  2. Verify: Search bar at top
  3. Verify: Table columns: Selector Key | URL | Success | Failures | Updated | Delete
  4. Verify: Fingerprints stored via API (Tests 2.1–2.2) are visible

  6. Test delete:

    • Verify: Confirmation appears ("Confirm" / "Cancel")
    • Click "Cancel" → nothing happens
    • Click trash again → click "Confirm" → fingerprint is deleted
    • Verify: Fingerprint disappears from list
  7. Test pagination: If >25 fingerprints, verify page controls work
  8. Test empty state: If all deleted, shows "No fingerprints found" with Fingerprint icon

This is the full realistic workflow: inject healing → build baselines → break selectors → verify auto-heal.

mkdir /tmp/e2e-heal-test && cd /tmp/e2e-heal-test
npm init -y
npm install @playwright/test
npx playwright install chromium

# In qabot REPL:
/heal inject /tmp/e2e-heal-test

Step 3: Verify in UI

  1. Go to http://localhost:3700/healing → Projects tab
  2. Verify: /tmp/e2e-heal-test appears as active injection
